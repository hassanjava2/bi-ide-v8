# Code-Free Idea Migration Policy

## الهدف
نقل قيمة الإصدارات السابقة إلى v8/v6 بدون نقل أي كود تنفيذي قديم.

## القاعدة الذهبية
- ممنوع نسخ/لصق أي كود من الإصدارات القديمة (v1..v7).
- المسموح فقط: الأفكار، المتطلبات، الأنماط المعمارية، معايير الجودة، وخرائط القرار.

## تعريف "فكرة" المسموح نقلها
- Capability statement: ماذا يجب أن يفعل النظام.
- Workflow statement: كيف تتدفق الخطوات من المدخل للنتيجة.
- Reliability policy: كيف نمنع التوقف/الضياع.
- Governance policy: صلاحيات، حدود، موافقات.

## تعريف "كود" الممنوع نقله
- أي ملف مصدر تنفيذي من القديم (JS/TS/Python/Rust/C#/SQL ...).
- أي snippet قابل للتنفيذ مأخوذ كما هو من القديم.
- أي config build/runtime من القديم بدون إعادة تصميم.

## آلية الحفظ بدون ضياع فكرة
1. استخراج الفكرة بصياغة محايدة عن اللغة.
2. ربطها بمصدرها (doc/section) فقط كمرجع مفهوم.
3. إعطاء الفكرة ID فريد + مالك + أولوية + KPI.
4. تحويلها إلى Task تنفيذي في v8.
5. اختبار قبول واضح قبل اعتبارها "محققة".

## بوابة الموافقة (Migration Gate)
لا تدخل أي فكرة للتنفيذ إلا إذا تحقق التالي:
- Problem واضح (لماذا هذه الفكرة مهمة الآن)
- KPI قابل للقياس
- Risk/Cost معلوم
- Owner محدد
- No-legacy-code check = PASS

## تعريف النجاح
- 0% code reuse من v1..v7
- 100% idea traceability من المصدر إلى التنفيذ
- 100% من الأفكار الحرجة لها Task + KPI + Owner
